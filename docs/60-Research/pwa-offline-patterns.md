# PWA Offline Patterns

> **MVP scope note (2026-05-18):** [[offline-mvp-scope-and-sync-strategy]] and
> [[../10-Architecture/09-Decisions/ADR-0020-hybrid-online-mvp-offline-ready]]
> supersede this note's original assumption that full offline-first
> singleplayer ships in MVP. This research remains the Phase-2+ reference for
> selective offline-first, export/import, migrations and sync.

Phase 1 research for an installable, offline-first football manager PWA.
Covers local save architecture, IndexedDB schema migrations, service worker
update flow, mutation queue / Background Sync, iOS/Android storage limits,
future SurrealDB cloud sync, and Playwright E2E test strategy.

Outputs feed [[ADR-0002-offline-first]] and [[ADR-0005-save-format]] (see the
two dedicated input sections at the end of this document).

## TL;DR

- Career saves live in **IndexedDB via Dexie 4** with a small set of
  schemafull tables (`saves`, `save_chunks`, `meta`). One IndexedDB database
  per browser profile, multiple career saves inside it.
- Every save row carries a `formatVersion` integer plus a Zod-validated
  envelope. **Forward-only migrations**, registered both as Dexie
  `version(N).upgrade(...)` blocks (for index/shape changes) and as
  in-process `migrateSave(envelope)` functions (for value-shape changes).
- Service worker is generated by Workbox (`generateSW`) and registered
  through `apps/web/public/sw-register.js`. For MVP we keep the current
  `skipWaiting: true` strategy, but the doc recommends moving to a
  **`workbox-window`-driven update prompt** before we ship interactive
  routes that hold unsaved state.
- App shell: `StaleWhileRevalidate` for assets, `NetworkFirst` for
  navigations, **never cache mutating HTTP** (POST/PUT/DELETE/PATCH).
  Background Sync is deferred (Chromium-only, brittle to test) and
  replaced by an IndexedDB-backed **outbox** that drains on `online`
  events plus on SW startup.
- iOS/Android in 2026: home-screen PWAs use full browser quota
  (~60% of disk on iOS 17+/Safari 17+), but only home-screen-installed
  apps are exempt from Safari ITP's 7-day eviction. We must
  `navigator.storage.persist()` early **and** prompt the user to
  install + export saves regularly.
- MVP cloud sync = **export/import JSON** only. When SurrealDB sync is
  introduced post-MVP, recommend a **server-authoritative LWW per save +
  per-document `updatedAt`** model with an op-log for replay; treat
  conflicts as user-visible "branches" rather than silent merges.

## 1. Scope and assumptions

In scope:

- Single-player career saves, fully playable offline.
- Installable PWA on Chromium (desktop + Android) and WebKit
  (iOS Safari + home-screen web apps).
- TanStack Start SSR shell + Workbox-built service worker
  (`apps/web/scripts/build-pwa.mjs`).
- Dexie 4.x as the only IndexedDB wrapper.
- SurrealDB optional cloud sync, **post-MVP**.

Out of scope (explicitly):

- Real-time multiplayer.
- Server-rendered career data — server is for static content + future sync.
- Real club/player identities (see [[ADR-0007-naming-schema]]).
- Migrating away from Dexie / IndexedDB.

Hard constraints from project docs:

- "Game state lives in IndexedDB via Dexie.js — never localStorage."
  ([AGENTS.md](../../AGENTS.md))
- Mutating HTTP responses must not be cached ([[08-Crosscutting]]).
- SurrealDB sync optional and post-MVP ([[ADR-0002-offline-first]]).

## 2. Recommended MVP save architecture

### 2.1 Storage layout

One Dexie database per origin, named `soccer-manager`.

| Table         | Primary key                  | Purpose                                                                    |
|---------------|------------------------------|----------------------------------------------------------------------------|
| `saves`       | `id` (ULID string)           | Save metadata + small fields: name, season, manager, club id, timestamps.  |
| `save_blobs`  | `[saveId+chunkIndex]`        | Large serialized state split into fixed-size chunks (gzip-compressed).     |
| `outbox`      | `++id`                       | Queued mutations destined for future cloud sync (drained on online).       |
| `meta`        | `key` (string)               | Singletons: `appVersion`, `lastOpenedSaveId`, `persistGranted`, locale.    |

Why split `saves` from `save_blobs`:

- Listing the save menu must be cheap. `db.saves.toArray()` should not
  deserialize hundreds of kilobytes of state.
- Chunking lets a single save grow well beyond the practical
  IndexedDB structured-clone size sweet spot (≤ ~1 MB per row stays
  fast; we cap chunks at 256 KB after gzip).
- Each chunk write is a separate IDB transaction commit, which limits
  damage from a mid-write crash and pairs cleanly with the corruption
  check in §3.4.

### 2.2 Save envelope (logical schema)

Every persisted state blob is a **`SaveEnvelope`** validated by Zod:

```ts
// pseudocode — see ADR-0005 input for normative shape
const SaveEnvelope = z.object({
  formatVersion: z.literal(1),         // bumped on breaking changes
  saveId: z.string().ulid(),
  appVersion: z.string(),              // semver of the app that wrote it
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
  schemaHash: z.string(),              // SHA-256 of the canonical TS schema
  payload: z.unknown(),                // versioned game state, see §3
})
```

`schemaHash` is generated at build-time from the TypeScript zod schema
and embedded in the bundle. It lets the loader detect "this save was
written by an unknown future build" before it tries to deserialize.

### 2.3 Write path

Atomic write of a save uses a single Dexie `'rw'` transaction across
both tables:

```ts
await db.transaction('rw', db.saves, db.save_blobs, async () => {
  await db.save_blobs.where('saveId').equals(id).delete()
  await db.save_blobs.bulkAdd(chunks)
  await db.saves.put({ ...meta, updatedAt: new Date().toISOString() })
})
```

Two reasons for the single-transaction approach:

1. IndexedDB transactions auto-commit and either fully apply or
   fully roll back, which gives us the atomicity that browser file
   APIs do not.
2. Dexie 4's transaction guarantees survive mid-transaction promise
   chains, which is what makes serializing → chunking → writing safe
   to express as an `async` function.

We never overwrite the previous save row in place: when the user
explicitly "Saves", we write a new envelope under the same `saveId`
plus an `autosave_n` rotation slot (see §3.5) so a corrupt write
cannot wipe the last good version.

## 3. IndexedDB / Dexie migration strategy

### 3.1 Two layers of versioning

We distinguish **schema** versioning from **payload** versioning.

| Layer                | What changes                                  | Mechanism                                        |
|----------------------|-----------------------------------------------|--------------------------------------------------|
| Dexie schema version | Tables, primary keys, indexes                 | `db.version(N).stores({...}).upgrade(tx => …)`   |
| Save payload version | Shape of the deserialized game state         | Pure functions `migrateSaveV1ToV2(payload)`      |

This split matters because adding an index never requires touching a
save payload, and changing a payload field never requires bumping the
IndexedDB version.

### 3.2 Dexie schema migrations

Per Dexie docs, the chained `version()` declarations are kept in code
so an upgrader knows the prior shape:

```ts
db.version(1).stores({
  saves: 'id, updatedAt',
  save_blobs: '[saveId+chunkIndex], saveId',
  outbox: '++id, createdAt',
  meta: 'key',
})

db.version(2).stores({
  saves: 'id, updatedAt, season',     // new index
}).upgrade(async tx => {
  await tx.table('saves').toCollection().modify(s => {
    s.season ??= deriveSeason(s)
  })
})
```

Rules we adopt:

- **Never delete** a previous `version()` block once it has shipped to
  any user. Append-only.
- `upgrade()` callbacks are pure with respect to network and clock —
  they may run on a phone in flight mode at first launch after update.
- Each `upgrade()` ships with a Vitest fixture: a snapshot IDB dump
  from the previous version is loaded into a fake Dexie via
  `fake-indexeddb` and asserted post-upgrade. **No upgrade ships
  without this test.**
- We accept Dexie 4's auto-version-detection but **still bump**
  `version(N)` on every schema change — the millisecond saved at boot
  is irrelevant; explicit version numbers make the upgrade graph
  reviewable in PRs.
- Dexie 4 supports downgrades for IDB shape but provides no
  callback. We therefore treat a downgrade (older app build opening
  a newer DB) as a hard error: the app shows a "save written by a
  newer version, please update" screen instead of partially loading.

### 3.3 Save payload migrations

Payload migrations are independent of Dexie:

```ts
type Migrator = (input: unknown) => unknown
const payloadMigrations: Record<number, Migrator> = {
  1: payload => migrateV1ToV2(payload),
  2: payload => migrateV2ToV3(payload),
}

function loadEnvelope(raw: unknown): SaveV3 {
  const env = SaveEnvelopeBase.parse(raw)
  let { payload, formatVersion } = env
  while (formatVersion < CURRENT_FORMAT_VERSION) {
    payload = payloadMigrations[formatVersion](payload)
    formatVersion += 1
  }
  return SaveV3.parse({ ...env, formatVersion, payload })
}
```

Properties:

- Migrations form a linear chain `v1 → v2 → v3 …`. **No branching.**
- Each migrator has 100% Vitest coverage on golden fixtures stored in
  `tests/fixtures/saves/v{N}/*.json`.
- Migrations run **on read**, never on background threads, never
  speculatively. The migrated envelope is written back to IDB
  on the next user-initiated save, not eagerly, so a botched migration
  cannot destroy an unmigrated original.
- We keep golden fixtures for every shipped `formatVersion`
  forever — they are the spec for "we will not break existing saves".

### 3.4 Corruption recovery

Recovery surfaces:

1. **Envelope parse fails** (Zod): show the user a "save corrupt"
   screen, offer to load the previous autosave slot, offer to
   download the raw bytes for support.
2. **Chunk count / hash mismatch**: each envelope stores a
   `chunkHash` (SHA-256 of the concatenated decompressed chunks).
   Loader recomputes it and treats mismatch as corruption.
3. **Storage quota exceeded mid-write**: IndexedDB rolls the
   transaction back — the previous save survives. We surface a
   `QuotaExceededError` to the user and refuse further saves until
   space is freed (delete other saves, export to disk).

### 3.5 Autosave rotation

To survive partial writes and bad migrations:

- Three rotating slots per career: `current`, `autosave_n-1`,
  `autosave_n-2`. Manual saves write to a separate `manual` slot.
- Rotation is a swap of pointer rows in `meta`, not a copy of blob
  data — cheap.
- "Restore previous" is a UI-exposed action, not an automatic
  fallback, so silent reverts cannot mask data loss.

## 4. Save format: import / export and corruption recovery

### 4.1 Export

User-initiated save export creates a single JSON file:

```jsonc
{
  "_format": "soccer-manager.save",
  "_formatVersion": 3,
  "_appVersion": "0.5.2",
  "_exportedAt": "2026-05-15T22:35:00.000Z",
  "envelope": { /* SaveEnvelope, payload inlined and decompressed */ }
}
```

Notes:

- Always inline-decompressed JSON. Gzip is an IDB storage detail, not
  a wire format. Users must be able to diff exports in a text editor.
- Filename pattern: `{managerSlug}-{season}-{ulid}.smsave.json`.
  The `.smsave.json` suffix lets browsers preview the file as text.
- A small SHA-256 of the canonical-JSON envelope is appended as
  `_envelopeHash` so re-import can detect transit corruption.

### 4.2 Import

```ts
const file = await pickFile()
const text = await file.text()
const parsed = ExportFile.parse(JSON.parse(text))   // Zod
verifyEnvelopeHash(parsed)
const envelope = await migrateEnvelope(parsed.envelope)
await writeSaveAtomically(envelope)
```

Failure modes are user-visible: parse failure, hash mismatch,
unknown future `formatVersion` (refuse), unknown `appVersion`
(warn but allow).

### 4.3 Why JSON, not SQLite / binary

- Inspectable, diffable, scriptable for power-users and tooling.
- Easier to write Vitest fixtures.
- Smaller surface area for "binary parser bug deletes save" failures.
- Compression is the wire-level concern of any future cloud sync, not
  of the file format.

## 5. Service worker update strategy

### 5.1 Current state

`apps/web/scripts/build-pwa.mjs` calls `workbox-build`'s `generateSW`
with:

- `clientsClaim: true`
- `skipWaiting: true`
- `cleanupOutdatedCaches: true`
- Routes:
  - `request.mode === 'navigate'` → `NetworkFirst` (cache `pages`)
  - `script | style | image | font` → `StaleWhileRevalidate` (cache `assets`)
- `navigateFallback: '/offline.html'`

`apps/web/public/sw-register.js` registers `/service-worker.js` 15 s
after `load` (or earlier on a custom `soccer-manager:register-sw`
event). It silently swallows registration errors so the bootstrap
shell stays usable.

### 5.2 What to keep

- **`StaleWhileRevalidate` for static assets** — well-suited to
  cache-busted hashed assets emitted by Vite.
- **`NetworkFirst` for navigations** — protects HTML freshness while
  still falling back offline.
- **Never cache mutating methods** — `runtimeCaching` in Workbox only
  matches GET by default; we keep that default and additionally
  pass `request.method === 'GET'` predicates explicitly to make
  intent obvious.
- **Late registration** — staying compatible with Lighthouse and
  giving hydration a head start is a defensible MVP choice.

### 5.3 What to change before interactive routes ship

The current setup uses `skipWaiting: true`, which means a new
service worker takes control as soon as it is installed. That is
acceptable for the static bootstrap shell but **dangerous** as soon
as routes hold unsaved game state, because the new SW's cached HTML
can swap underneath the running app and hashed assets may go missing.

Recommendation: switch to the **`workbox-window` update prompt
pattern** (Chrome docs):

```ts
// client
import { Workbox } from 'workbox-window'
const wb = new Workbox('/service-worker.js')
wb.addEventListener('waiting', async () => {
  if (await ui.promptForUpdate()) {
    wb.addEventListener('controlling', () => location.reload())
    wb.messageSkipWaiting()
  }
})
wb.register()
```

```ts
// service worker (still generated by generateSW with skipWaiting: false)
addEventListener('message', e => {
  if (e.data?.type === 'SKIP_WAITING') self.skipWaiting()
})
```

Concretely this means:

1. Flip `skipWaiting: true` → `false` in `build-pwa.mjs`.
2. Replace inline registration in `sw-register.js` with a
   `workbox-window`-based registrar gated on hydration.
3. Add a "Reload to update" toast component using shadcn `Toast`.
4. Before showing the prompt, **flush the outbox + autosave** so an
   accepted reload never loses unsaved state.

### 5.4 Versioning the SW build

We treat the contents of `service-worker.js` as immutable per build.
The script tag includes a hashed query (`?v=${appVersion}`) so a
new app version always triggers an SW update check, even on browsers
that skip the byte-by-byte SW comparison when the URL is unchanged.

### 5.5 Update telemetry (post-MVP)

When a user accepts an update, log
`{ from: prevAppVersion, to: nextAppVersion }` so we can see how long
old SWs linger in the field. This is the single most useful signal
for catching update-stuck PWAs, especially on iOS.

## 6. Background Sync and the mutation queue

### 6.1 Why we do not adopt the Background Sync API for MVP

The Background Sync API is well-designed but has three properties
that make it the wrong primitive for an offline-first single-player
game in 2026:

- **Chromium-only.** Safari and Firefox have no implementation.
  Workbox's fallback is "replay on next service worker startup",
  which requires the page to be open — i.e. the same trigger as
  our own `online` event.
- **Designed for HTTP retries**, not for a domain-level mutation
  log. Replays send the original `Request`; our future
  cloud-sync mutations are domain operations like
  `applyTransfer(playerId, ...)`, not REST calls.
- **Unfriendly to test.** Workbox's own docs note that DevTools
  "offline" does not affect SW fetches and that you must trigger
  sync events manually from DevTools — Playwright cannot do this.

### 6.2 Outbox pattern (recommended for MVP)

Mutations destined for future cloud sync are appended to the
`outbox` Dexie table:

```ts
interface OutboxEntry {
  id?: number          // auto-increment
  saveId: string
  op: string           // domain op name, e.g. 'transfer.confirm'
  args: unknown        // structured-cloneable
  createdAt: string    // ISO
  attempts: number
  lastError?: string
}
```

Drain triggers, in priority order:

1. `window.addEventListener('online', drainOutbox)`.
2. SW `activate` posts a `DRAIN_OUTBOX` message to clients.
3. Manual "Sync now" UI action.
4. On every successful save commit (cheap, in-process).

The drainer has the same retry-with-exponential-backoff semantics
as Workbox's `BackgroundSyncPlugin`, but operates on **domain ops**
rather than serialized Requests. This keeps the conflict-resolution
section (§9) implementable.

### 6.3 When to revisit Background Sync

If we later add **per-action HTTP idempotency keys** for cloud
sync, the Workbox `BackgroundSyncPlugin` becomes attractive for
the leaf transport layer (POST that retry until 2xx), wrapped
around the same outbox. Until then: domain-level outbox only.

## 7. iOS / Android storage and install behavior

### 7.1 Quotas (2026 baseline)

| Browser            | Best-effort quota                  | Persistent quota               |
|--------------------|-----------------------------------|--------------------------------|
| Chromium (desktop + Android) | up to ~60% of total disk    | up to ~60% of total disk       |
| Firefox            | 10% of disk or 10 GiB (group cap) | up to 50% of disk, cap 8 TiB   |
| Safari iOS 17+ / macOS Sonoma+ (browser app or home-screen PWA) | up to ~60% of disk | same; persistent via heuristic |
| Safari embedded WKWebView (in-app) | up to ~15% of disk      | n/a — best-effort only         |

Exact numbers are upper bounds. Browsers cap further to avoid
fingerprinting, so always handle `QuotaExceededError`.

### 7.2 Eviction triggers

- **LRU under storage pressure** — least-recently-used origins
  evicted first; persistent origins skipped (Chromium, Firefox,
  Safari).
- **Overall-quota eviction** (Chromium ~80% of disk, Safari ~80%
  for browser apps): same LRU rules.
- **Safari ITP proactive eviction**: 7 days of no user
  interaction → script-created data (IDB, Cache, OPFS, localStorage)
  deleted. Cookies set by the *server* are exempt.
- **Home-screen PWAs are exempt** from the 7-day ITP eviction —
  this is the single most important reason to push the
  "Add to Home Screen" flow.

### 7.3 What we must do

1. Call `navigator.storage.persist()` immediately after the user
   creates their first save. If it returns `false`, surface a UI
   chip "Saves may be evicted — install the app for full
   persistence". WebKit's heuristic grants persistence
   automatically once installed to the Home Screen.
2. Show free-space telemetry from `navigator.storage.estimate()`
   on the save management screen.
3. On every save success, prompt to **export** if the user has not
   exported in N days. Treat exports as the durable backup, not
   IndexedDB.
4. Detect "browser-only Safari" sessions and show a one-time
   educational toast: "Add to Home Screen so iOS keeps your
   saves longer than 7 days."
5. **Do not** rely on `localStorage` for anything (also enforced
   by the always-applied workspace rule).

### 7.4 Install behavior quirks

- **iOS**: `beforeinstallprompt` not supported; install is a manual
  Safari Share-Sheet flow. Detect standalone mode with
  `window.matchMedia('(display-mode: standalone)')`.
- **iOS pre-17.4**: third-party browsers are WebKit underneath, so
  the Safari quotas apply. Post-17.4 EU users may use alternative
  engines with their own policies — assume Safari rules unless
  detected otherwise.
- **Android Chrome**: `beforeinstallprompt` works; we capture and
  defer it, and re-prompt on the save screen after first save.
- **Lighthouse `installable-manifest`** audit must stay green in
  CI. The bootstrap shell deliberately defers SW registration to
  protect that score; that constraint is preserved.

### 7.5 Risks called out

- **Safari embedded WKWebView (e.g. opened from another app's
  in-app browser) gets 15% of disk and no persistent mode.**
  We must detect this and refuse to create a new save in that
  context, redirecting users to open in Safari.
- **`navigator.storage.estimate()` is approximate** and pads
  cross-origin caches; do not use it as the source of truth for
  blocking writes.

## 8. Future cloud sync — conflict resolution

This section frames the post-MVP SurrealDB sync. None of it is
implemented in MVP; the outbox in §6 is the seam.

### 8.1 The problem

A career save is a **single-writer, multi-device** document. The
typical conflict is "I played for 30 minutes on the train (offline,
phone), then opened on my desktop where I had played 10 minutes
the night before". This is materially easier than collaborative
multi-writer editing.

### 8.2 Options surveyed

| Option                                  | Suitability                                     | Notes                                                                                   |
|-----------------------------------------|------------------------------------------------|-----------------------------------------------------------------------------------------|
| Last-write-wins on whole save           | OK for MVP cloud, lossy on conflicts            | Simple, but discards 30 minutes of phone play silently. Not user-acceptable as default. |
| Per-document LWW + `updatedAt`          | Good for fine-grained domain entities (player, club) but a single save is one document in our model. | Useful only if we re-shape the save into many SurrealDB records. |
| Operational log (op-log) replay         | Strong fit for our outbox model.                | Each device appends ops; server defines the canonical order; clients replay.            |
| CRDTs (Yjs / Automerge)                 | Overkill for single-writer; expensive payloads. | Useful when we add features like collaborative tactics boards — not for career state.   |
| Manual "branches"                        | Low-tech, user-honest.                         | When two devices diverge, surface both saves and let the user pick or fork.             |

### 8.3 Recommendation

For SurrealDB cloud sync (post-MVP), adopt a **two-tier model**:

1. **Per-save server-authoritative LWW** keyed on
   `(saveId, lastSyncSeq)`. The server rejects writes whose
   `lastSyncSeq` is older than the server's, returning the newer
   envelope. The client then prompts the user with the diff:
   "Phone version is 30 min ahead of cloud. Keep phone, keep cloud,
   or fork into a second save."
2. **Op-log replay** for each save under the hood. The outbox
   (§6.2) is already an op-log; the cloud just appends and
   replays it on the canonical version.

Why this stack:

- It's honest with the user: silent merges of game state cause
  trust collapse the first time a transfer "disappears".
- It maps cleanly to SurrealDB's record links + RELATE model
  ([AGENTS.md](../../AGENTS.md)) and to per-save `DEFINE TABLE` rules.
- It does not require shipping a CRDT runtime in the bundle.
- It preserves the option to upgrade specific sub-trees
  (e.g. tactics) to CRDTs later without redoing the transport.

### 8.4 Anti-patterns to avoid

- **Whole-save LWW with no user prompt.** Will lose hours of play.
- **Vector clocks across browser instances.** We have no reliable
  device identity; rotating IDs per install + user re-install on
  iOS make this brittle.
- **Eager merging of payload fields** without semantic awareness
  (e.g. summing numeric fields). Always prompt or defer.

## 9. Offline E2E test strategy with Playwright

### 9.1 What Playwright can and cannot do

- **Can:** `await context.setOffline(true)` flips the page's
  network state and fires the `offline` event in the page.
- **Cannot reliably:** make the **service worker** see the offline
  state. Per Playwright's own docs and well-known issue
  (open since 2020), `setOffline()` is per page target, and the SW
  target stays online — it will continue to serve cached pages and
  perform new fetches normally. WebSockets and keep-alive HTTP
  sockets also stay open across the flip.

### 9.2 Patterns we adopt

1. **Always `await navigator.serviceWorker.ready`** inside the
   page before flipping offline. The runner is faster than the
   browser at registering the worker.
2. **Precache deliberately**: if a test asserts an offline page
   load, the page must already be in the Workbox `pages` cache
   from a prior visit in the same test. Use a navigation step
   to warm the cache before `setOffline(true)`.
3. **Prefer `setOffline()` over `route()` interception** for SW
   tests. `route()` runs in Node and never sees requests the SW
   handles internally.
4. **Test the outbox via app-level UI**, not via DevTools
   "Sync" buttons (Playwright cannot click them):
   - Go offline → trigger mutation in UI → assert outbox has 1
     entry (via test-only API or a debug overlay).
   - Go online → assert outbox drains, mutation is "synced".
5. **Service-worker code paths get unit + integration tests in
   Vitest with a Workbox test harness**; Playwright covers
   end-to-end PWA install + reload-to-update flows.

### 9.3 Offline E2E test matrix (MVP)

| Scenario                                              | Tooling                                     |
|-------------------------------------------------------|---------------------------------------------|
| Fresh visit → offline → reload → app shell renders    | Playwright `setOffline(true)` + cache warm  |
| Save → offline → reload → save still loads            | Playwright + Dexie via in-page eval         |
| Migration v(N-1) → v(N) on first launch after update  | Vitest + `fake-indexeddb` + golden fixtures |
| Outbox queues mutations offline and drains online     | Playwright with debug API on `window`       |
| Quota exceeded mid-write → previous save survives     | Vitest with mocked Dexie throwing           |
| SW update prompt → user accepts → reload              | Playwright by deploying two app versions in serial test runs |
| Lighthouse `installable-manifest` audit               | CI Lighthouse step (already in plan)         |
| iOS install + offline launch                          | Out of automation; manual smoke checklist   |

### 9.4 Things we explicitly do not test in CI

- `beforeinstallprompt` install acceptance (per project rule:
  "Install-prompt testing is unsupported in Playwright; use
  Lighthouse `installable-manifest` audit").
- True cross-device sync (no infrastructure yet).
- Real Background Sync `sync` events (DevTools-only trigger).

## 10. Open questions

These do not block MVP but should be resolved before cloud sync ships:

- Do we encrypt save exports at rest? If yes, who owns the key?
  (Likely: optional symmetric passphrase, never default-on.)
- Do we ship a "save format linter" CLI in `packages/`? Useful for
  community shared saves, post-MVP.
- How do we evict old autosave slots when a save grows past the
  origin quota? Current plan: refuse autosave + surface UI; alternative
  is silently rotate. Recommend "refuse + UI" for trust.
- Should we mirror saves to OPFS as an additional durable layer?
  OPFS shares the same origin quota and the same eviction rules,
  so it adds **no** persistence guarantee — recommendation: **no**,
  not worth the second code path.

## 11. Direct input for ADR-0002 — Offline-first strategy

> Drop-in section the ADR can lift verbatim under "Decision" /
> "Consequences" / "Status quo".

### Decision (proposed)

- Persist all gameplay state in **IndexedDB via Dexie 4** in a single
  origin database `soccer-manager`. Never use `localStorage`.
- Always opt into persistent storage via
  `navigator.storage.persist()` after the first save commit, and
  expose `navigator.storage.estimate()` in the save management UI.
- Service worker is generated by Workbox (`generateSW`) and uses:
  - `NetworkFirst` for navigation requests with `/offline.html`
    fallback,
  - `StaleWhileRevalidate` for hashed static assets,
  - **never** caches non-GET requests.
- For MVP, mutations destined for future cloud sync are queued in
  an **IndexedDB-backed outbox** (Dexie table), not via
  Background Sync API. Outbox drains on `online`, on SW activation,
  on save commit, and via a manual UI action.
- Switch the SW from `skipWaiting: true` to a **`workbox-window`
  update prompt** before any route holds unsaved interactive state.
  Until then, current behavior is acceptable for the static
  bootstrap shell.
- iOS users are explicitly steered to "Add to Home Screen" because
  only home-screen PWAs are exempt from Safari ITP's 7-day eviction.
- SurrealDB cloud sync is **out of scope for MVP**. The outbox is
  the seam; see §8 for the chosen conflict-resolution model when
  sync is added.

### Consequences

- Gameplay must work fully without network access, including all
  match simulation and save/load. No route loader may depend on
  a server function for game data after first save.
- Save migrations are forward-only and tested. Downgrades are a
  hard-error UX, not a silent partial load.
- We accept that Safari embedded WKWebView is too constrained for
  a full save; the app refuses to create new saves in that
  context.
- We accept the cost of building our own outbox in exchange for
  a sync-ready, testable, cross-browser primitive.

### Status

Proposed — pending ADR-0002 finalization (AKOM-121).

## 12. Direct input for ADR-0005 — Save format and versioning

> Drop-in section the ADR can lift verbatim.

### Decision (proposed)

- One **`SaveEnvelope`** Zod schema is the source of truth for every
  persisted save, both in IndexedDB and in exported JSON files.
- The envelope carries:
  - `formatVersion: integer` — bumped on every breaking payload
    change.
  - `saveId: ULID`.
  - `appVersion: semver` — recorded for diagnostics; not used to
    gate loading.
  - `schemaHash: SHA-256` of the canonical TS schema at build time.
  - `createdAt`, `updatedAt`: ISO-8601 datetimes.
  - `payload: unknown` — versioned game state.
  - `chunkHash: SHA-256` of the concatenated decompressed payload
    bytes (in IDB form), used to detect chunk corruption.
- Storage layout is `saves` (metadata) + `save_blobs` (gzip-compressed
  fixed-size chunks, ≤ 256 KB each), written atomically inside one
  Dexie `'rw'` transaction.
- Migrations live in two layers:
  - Dexie schema migrations, append-only `db.version(N)` blocks
    with `upgrade()` callbacks. Every shipped block remains in
    the codebase forever.
  - **Payload migrations**: pure functions
    `migrateV(N)ToV(N+1)(payload: unknown): unknown`, chained
    linearly on read, writing back only on next user-initiated
    save. Golden fixtures in `tests/fixtures/saves/v{N}/` are the
    spec.
- Export format is a single `.smsave.json` file with the
  decompressed envelope inlined plus an `_envelopeHash` of its
  canonical-JSON form.
- Autosave uses three rotating slots per career; manual save uses a
  separate slot. The previous good save is never overwritten in
  place.
- An "unknown future `formatVersion`" on import is a **refuse**;
  an "unknown `appVersion`" is a **warn**. A `chunkHash` mismatch
  is a **refuse + offer previous autosave**.

### Consequences

- Every breaking save change ships with a Vitest migration test
  loading a v(N-1) fixture and asserting v(N) shape. CI gate.
- All gameplay state must be **structured-cloneable** and
  serializable to JSON without information loss. No `Date` objects
  in payloads — ISO strings only. No `Map`/`Set` in payloads
  except via explicit (de)serializers.
- The save format is the public contract. Renaming a payload field
  requires a migration, even when the type is unchanged.
- Compression is an implementation detail of IDB storage and never
  appears in exports.
- Saves remain inspectable, diffable, and trivially scriptable for
  community tooling and bug reports.

### Status

Proposed — pending ADR-0005 finalization (AKOM-117).
