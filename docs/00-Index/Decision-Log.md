# Decision Log

ADR index with status and lineage. Implement only from `accepted` ADRs or
`current binding` research notes explicitly promoted here. Draft/proposed ADRs
are planning context only.

ADRs decide **how the game is built**. Game-design notes and GDDRs in
[[../50-Game-Design/README]] decide how the game works; ADRs must not contradict
an `approved` game-design record. Chain: research -> game design -> ADRs ->
implementation. If approved game-design records conflict, stop and supersede one
explicitly before implementation.

Redis-based progressive throttling (per-account 0/2/10/30/120/600/1800 s ladder + soft 1 h lock @ ≥ 50 failures/hour; per-IP 30/5 min token bucket; per-/24 200/5 min). Password-reset throttle (3 / 15 min + 10 / 24 h per email + outbound SMTP cap); signup throttle (5 / 15 min per IP + 3 / 24 h per email). **No external IdP (Google / Apple / Discord / Microsoft / GitHub / Meta / X) at MVP** — `user_identity` schema + OAuth abstraction provisioned now so post-MVP addition is additive; when added, authorization-code + PKCE only, scopes ≤ `openid email`, linking by `(provider, sub)` not email-equality. **No CAPTCHA at MVP** — rate-limit + email verification only; stage-1 abuse response is self-hosted mCaptcha behind feature flag, stage-2 Friendly Captcha (EU-managed); reCAPTCHA / hCaptcha / Cloudflare Turnstile explicitly rejected on GDPR posture. Libraries: `@simplewebauthn/server` + `@simplewebauthn/browser`, `argon2` (libsodium-backed), `openid-client` for future social IdP; **`passport.js` rejected**. Compliance: full ASVS v5.0 V6 L2 mapping (33 controls) + NIST SP 800-63B AAL2 anchors (passkey alone qualifies via UV; password + TOTP also qualifies). Anchors on [[../60-Research/threat-model]] (F1) §1.3, §3, §4.1, §4.6, §5; refines ADR-0005 §3 by adding the bootstrap endpoint contract; binds inputs for **F3 Session management** (Redis schema, device list, "log out everywhere" runbook), **F5 Account recovery** (stable account-master-key envelope for save-key continuity across rotation), **F6 GDPR compliance** (DSAR including `auth_events`, DPIA on `accountSecret` + WebAuthn storage), **F12 Rate limiting** (edge WAF + mCaptcha graduation). - [[../30-Implementation/session-management]] — Locks the session-lifecycle implementation (F3, 2026-05-18). - [[../30-Implementation/account-recovery]] — Locks the account-recovery + key-rotation implementation (F5, 2026-05-18). **Stable inner master key `K`** (256-bit AES-GCM, generated once per account at email verification, non-extractable on every device, never seen by server) + **canonical user-level envelope** `Env_user = AES-GCM-256(K, KEK_user)` with `KEK_user = PBKDF2-SHA256(accountSecret, userSalt, 600_000, 256)`. **Multi-device continuity has NO cross-device protocol**: rotation re-wraps `Env_user` once; other devices fetch the new `Env_user` on re-login via F2 §5.3 bootstrap and unwrap `K` with the new `accountSecret`-derived `KEK_user`. Per-device envelopes (`Env_device`) demoted to optional offline-cache optimisation. **10 recovery-code envelopes** `Env_recovery_i = AES-GCM-256(K, KEK_recovery_i)` per user — independent recovery path that survives total device + password loss; recovery-code KDF uses the same 600 000-iteration PBKDF2. **Portable-export envelope** `Env_portable` from user passphrase, survives `accountSecret` rotation by design (independent KDF input). Versioned envelope wire format with full AAD binding (envelope-magic + version + wrapMode + wrapTargetId + KDF/AEAD discriminators + algorithm params + salt + keyId) preventing transplant and rollback attacks. **Web Crypto mechanics**: `crypto.subtle.generateKey({extractable: true}) → wrapKey('raw', ...) → re-import as extractable=false`; `wrapKey/unwrapKey` semantics ensure `K` bytes never enter JS heap on subsequent device launches; `CryptoKey` survives `IndexedDB.put()` round-trip with `extractable: false` preserved on Chrome / Edge / Firefox / Safari 16+. **Atomic rotation algorithm** with Redis `rotation_lock:<user_id>` (60 s TTL), idempotency-key replay protection, SurrealDB transaction wrapping `accountSecret` update + `env_user` swap + `account_secret_version++` + `envelope_version` bump + `token_version++` + `device.env_device` cache wipe + outbox `auth.account_secret_rotated` emission. **Three rotation triggers**: password change (logged-in session, F2 §7.1), recovery-code use (login screen, F2 §6.2 with mandatory regen of all 10 codes + `userSalt` rotation), "Sign out everywhere AND rotate security key" (F3 §9.5). **Email password reset DOES rotate `accountSecret`** (high-signal compromise event per industry guidance) but does NOT regenerate recovery codes (their envelopes still wrap unchanged `K`). **F2 → F5 lazy migration** on first F5-enabled login per user: derive `K`, generate `userSalt`, wrap `Env_user`, regenerate the 10 recovery codes with new envelopes, one-shot re-encryption of existing IndexedDB saves under `K` inside the migration transaction (~1-2 s for a typical user); idempotent; backwards-compatible offline (existing direct-derive saves remain readable until first online session). **Three recovery flows**: email password reset (single-use 15-min token, generic anti-enumeration response, full session revoke, accountSecret rotation), recovery-code use (login-screen entry, Argon2id verify, mandatory regen of 10 fresh codes, `userSalt` rotation), "Sign out everywhere AND rotate security key" (Settings → Security button). **Cannot-recover cliff confirmed** (F2 Q4 + Q4 research): if user loses passkey + password + all 10 recovery codes, account is unrecoverable by design; portable-export with remembered passphrase is the only escape path. **10-row attack mitigation matrix** (reset-email intercept, recovery-code phishing / stuffing / replay, oracle timing, email-change → reset chain, server compromise, rollback, envelope transplant). **Server-side schema**: `user.env_user` + `user.user_salt` + `user.envelope_version` + `user.account_secret_version` + 10 × `user_credential(kind=recovery_code).payload.{salt, code_hash, env_recovery}` + optional `device.env_device` cache. **Future-proof extensions provisioned now**: `envelopeVersion = 2` Argon2id KEK for portable export (post-MVP UI ship); `envelopeVersion = 3` HPKE / RFC 9180 KEM for post-quantum migration; `wrapMode = 'shared_save'` for Phase-2 cloud MP per-member content keys; `wrapMode = 'device_bound'` for RFC 9449 DPoP hardware-bound wrap. **Compliance**: full mapping to NIST SP 800-130 CKMS + NIST SP 800-63B rev. 4 §6 authenticator binding/loss + NIST SP 800-132 password-based KDF + NIST SP 800-38D AES-GCM + NIST SP 800-57 Pt. 1 key lifecycle + OWASP ASVS v5 V6 + V11 + RFC 8018 / 5990 / 9180 / 9449 + W3C WebCrypto Level 2 §7. Anchors on F1 ([[../60-Research/threat-model]]) + F2 ([[../30-Implementation/auth-flows]]) + F3 ([[../30-Implementation/session-management]]); **closes F2 FU-1** (account-master-key envelope wire format + migration) and **closes F3 FU-7** (stable account-master-key envelope decoupling rotation from device-backup save key); binds inputs for **F6 GDPR compliance** (DSAR + DPIA on K + Env_user + recovery-code storage, FU-8 + FU-9), **F4 multi-device sync** (post-MVP cloud MP; `wrapMode = 'shared_save'` provisioned). Surfaces 6 product-owner Q&A questions (canonical Env_user architecture, one-shot save re-encryption during migration, userSalt rotation on recovery-code only, password reset rotates accountSecret, recovery codes regenerated only on recovery-code use, full AAD layout). 9 follow-up tasks (FU-1..FU-9) anchored to E10/E11/F4/F6/post-MVP. Status `current binding`.
